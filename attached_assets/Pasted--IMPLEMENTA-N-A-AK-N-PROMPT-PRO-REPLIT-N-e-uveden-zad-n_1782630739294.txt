# IMPLEMENTAČNÍ A AKČNÍ PROMPT PRO REPLIT

Níže uvedené zadání proveď nad aplikací **Site-Logbook / Stavba pro Modvolt s.r.o.**

Nejde o návrh ani obecný code review. Proveď změny v kódu, migracích, testech a deployment workflow, aplikaci spusť, používej ji přes skutečné UI a dolož PASS/FAIL výsledky.

## 1. Role a pracovní režim

Jsi senior full-stack vývojář, databázový inženýr, QA a produktový UX engineer.

Při implementaci a testování se přepínej mezi těmito uživateli:

1. Technik na stavbě:
   - pracuje na mobilu;
   - spěchá;
   - potřebuje rychle najít zakázku;
   - zapisuje krátké české poznámky;
   - přidává materiál, čas a fotografie;
   - nesmí přijít o rozepsaná data.
2. Administrátor:
   - spravuje zákazníky, zakázky, lidi, stroje, přístupy a oprávnění;
   - potřebuje odlišit rozpracované, hotové, problémové a nevyfakturované;
   - potřebuje bezpečné zálohy a srozumitelné technické zdraví aplikace.
3. Fakturant:
   - zpracovává hotové práce, dodavatelské faktury a dodací listy;
   - kontroluje AI vytěžení;
   - opravuje řádky a ceny;
   - potřebuje jasně vidět, co čeká na ruční kontrolu.
4. Majitel:
   - potřebuje do několika sekund vidět rizika, nevyfakturovanou práci, marži a stav záloh;
   - nesmí být uveden v omyl falešnou nulou nebo falešně prázdným seznamem.

## 2. Výslovně mimo rozsah

Neimplementuj ani nenavrhuj:

- **mobilní prioritizaci/přeskupení fakturačního rozcestníku**;
- nový modul „Rychlý záznam technika“.

Současnou strukturu mobilního fakturačního rozcestníku ponech. Oprav jeho funkčnost, chyby, překryvy a datovou spolehlivost, ale nepřeskupuj karty podle priorit.

## 3. Ochrana dat a bezpečnostní pravidla

Dodrž bez výjimky:

- nemaž existující uživatelské přílohy, fotografie, doklady, faktury ani přístupové údaje;
- nepoužívej `git reset --hard`, `push --force` ani `drizzle-kit push --force`;
- nespouštěj produkční migraci bez čerstvé zálohy a ověření její čitelnosti;
- nepřepisuj historii skladových pohybů;
- sklad je append-only ledger, opravy se provádějí stornem/kompenzačním pohybem;
- ceny již provedených výdejů musí zůstat historicky stabilní;
- neukládej hesla, PINy, API klíče ani přihlašovací údaje do zdrojového kódu, testů nebo reportů;
- všechny E2E testy musí používat proměnné prostředí;
- testovací data označ prefixem `E2E_` nebo `AUDIT_` a po testu je bezpečně uklidit;
- při blockeru migrace aplikaci raději nespouštěj, než aby běžela proti nekompatibilnímu schématu.

## 4. Potvrzený výchozí problém

Live development měl následující stav:

| Kontrola | Výsledek |
|---|---:|
| `GET /api/activities` | 200 |
| `GET /api/billing/ai-extraction` | 200 |
| `GET /api/billing/document-linking` | 200 |
| `GET /api/jobs` | 500 |
| `POST /api/jobs` | 500 |
| `GET /api/warehouse-movements` | 500 |
| `POST /api/warehouse-items/:id/movements` | 500 |
| `GET /api/billing/unbilled` | 500 |
| `GET /api/billing/settings` | 500 |

Kořenová příčina je drift databázového schématu:

- aplikace očekává `jobs.short_name`;
- sklad očekává `warehouse_movements.cost_price_at_time`;
- fakturace očekává `billing_settings.margin_alert_threshold_percent`;
- nová revize obsahuje migrace až do `0033`;
- původní Replit `postMerge` používal chybný filtr `pnpm --filter db push`;
- `push` je interaktivní dev operace a nesmí nahrazovat verzované migrace.

## 5. Povinné pořadí implementace

Pracuj po fázích. Po každé fázi spusť relevantní testy. Nepřeskakuj na kosmetiku, dokud nejsou P0 workflow funkční.

# FÁZE A - P0 STABILIZACE A MIGRACE

## A1. Vynucení parity databáze

### Problém

API se může spustit proti starému schématu a až při práci uživatele vracet 500.

### Proč to vadí

Dashboard vypadá funkčně, ale technik nemůže založit zakázku, sklad nemůže zapsat pohyb a fakturant nevidí hotovou práci.

### Implementace

1. Zkontroluj:
   - `scripts/post-merge.sh`;
   - `.replit`;
   - `artifacts/api-server/package.json`;
   - `artifacts/api-server/src/migrate.ts`;
   - `lib/db/src/migrate.ts`;
   - `lib/db/migrations/meta/_journal.json`.
2. Replit musí používat:

```text
pnpm --filter @workspace/db run migrate
```

3. API development start musí spustit migraci před build/start.
4. Produkční API musí spustit `dist/migrate.mjs` před `dist/index.mjs`.
5. Migrátor musí:
   - použít pouze committed SQL migrace;
   - ověřit všechny journal tagy;
   - při chybějícím tagu skončit nenulovým kódem;
   - zalogovat expected/applied/latest/missing;
   - zabránit startu API.
6. `postMerge` timeout nastav minimálně na 120 sekund.
7. Nepoužívej interaktivní `push`.

### Migrace, které musí být zahrnuté

- `0025_tan_cobalt_man`;
- `0026_tidy_steel_serpent`;
- `0027_flashy_nicolaos`;
- `0028_backfill_cost_price_at_time`;
- `0029_backfill_purchase_price_history`;
- `0030_old_rafael_vega`;
- `0031_margin_alert_threshold`;
- `0032_job_visits`;
- `0033_block_legacy_billed_activities`.

### Acceptance

- API proti DB s chybějící migrací odmítne start;
- API proti aktuální DB startuje;
- log obsahuje úplnou migration parity;
- žádná stávající data nejsou smazána.

## A2. Zprovoznění zakázek

### Problém

Seznam i vytvoření zakázky vracejí 500.

### Implementace

- oprav schéma/migraci, neobcházej chybějící sloupec v UI;
- zachovej `short_name` kompatibilní se starými řádky;
- ověř seznam, filtry, detail, vytvoření, aktualizaci a smazání testovacího záznamu;
- dvojklik na `Uložit` nesmí založit dvě zakázky;
- pending stav musí deaktivovat obě save akce;
- chyba musí zachovat obsah formuláře.

### Acceptance

- `GET /api/jobs` = 200;
- `POST /api/jobs` = 201;
- dlouhý český název a poznámka přežijí reload;
- množství `5,5` a cena `42,90` se uloží správně;
- dvojklik nevytvoří duplicitu.

## A3. Zprovoznění skladu a cen v čase

### Problém

Položka jde vytvořit, ale příjem/výdej a kniha pohybů selhávají.

### Implementace

- aplikuj `cost_price_at_time` migrace a backfill;
- zachovej append-only ledger;
- výdej musí uložit nákupní cenu platnou v okamžiku pohybu;
- pozdější změna ceny nesmí přepsat historickou marži;
- přidej bezpečné storno posledního povoleného pohybu;
- prázdná nebo chybová kniha nesmí zůstat v nekonečném skeletonu;
- při chybě zobraz `Zkusit znovu`.

### Acceptance

- `GET /api/warehouse-movements` = 200;
- příjem = 201;
- výdej na zakázku = 201;
- storno vytvoří korekční záznam;
- skladová hodnota, stav a marže se přepočítají;
- historie původního pohybu zůstane neměnná.

## A4. Spolehlivá fronta nevyfakturovaných prací

### Problém

Fakturant nevidí hotové práce a může přijít o peníze.

### Implementace

- `GET /api/billing/unbilled` musí vracet zakázky i dokončené dlouhodobé akce;
- fakturovanost určuj podle `invoice_source_links` a necanceled faktury;
- kosmetický `billing_status` nesmí být autoritativní;
- zákazník s pouze dokončenou akcí se musí objevit;
- ukaž důvody, proč položku nelze fakturovat:
  - chybí zákazník;
  - chybí cena;
  - chybí povinné fakturační údaje;
  - již je propojena s konceptem/fakturou.
- návrh faktury nesmí stejnou práci zahrnout dvakrát.

### Acceptance

- `GET /api/billing/unbilled` = 200;
- dokončená akce se zákazníkem a materiálem je viditelná;
- vytvoření konceptu ji odstraní z volné fronty;
- zrušení faktury obnoví správný stav podle pravidel;
- součty zakázek a akcí sedí s detailem.

## A5. Validace dokladů

### Problém

Live server přijal `.txt` jako účetní doklad, přestože UI deklaruje PDF/foto/ISDOC/XML/ZIP.

### Implementace

- sjednoť allowlist klienta a serveru;
- povol:
  - PDF;
  - JPEG/JPG;
  - PNG;
  - WEBP;
  - GIF;
  - HEIC/HEIF;
  - XML/ISDOC;
  - ISDOCX;
  - ZIP obsahující podporované typy.
- odmítni TXT, CSV, Word, Excel a jiný balast;
- kontroluj příponu, MIME i skutečnou signaturu;
- ZIP rozbal bezpečně bez path traversal a bez rekurzivních archivů;
- nepodporovaný přímý soubor odmítni ještě před requestem;
- server musí přesto nezávisle vrátit 415 při obejití klienta;
- MIME pro HEIC/ZIP odvoď z přípony, pokud ho telefon neposkytne.

### Acceptance

- `.txt` nevyvolá upload request;
- přímý API upload `.txt` = 415;
- falešný `.jpg` s textovým obsahem = 415;
- legitimní PDF, HEIC a ISDOCX projdou;
- duplicitní dokument vyvolá kontrolovaný 409 workflow.

## A6. Zálohy databáze a příloh

### Problém

Existence endpointu záloh není důkaz obnovitelnosti.

### Implementace

- vytvoř automatické DB zálohy;
- zahrň objektové úložiště nebo pro něj definuj samostatnou snapshot politiku;
- zobraz:
  - poslední úspěšnou zálohu;
  - velikost;
  - checksum;
  - poslední chybu;
  - poslední úspěšný restore test.
- vytvoř dokumentovaný restore postup pro izolované prostředí;
- nikdy nespouštěj restore nad živou DB bez explicitního potvrzení.

### Acceptance

- ruční záloha je vytvořena a stažitelná;
- checksum sedí;
- izolovaný restore obnoví zákazníky, zakázky, faktury a vazby na přílohy;
- výsledek restore testu je auditovaný.

# FÁZE B - POTVRZENÉ ÚZKÉ OPRAVY UI A API

Pokud tyto změny v aktuálním branchi již existují, nereimplementuj je duplicitně. Zkontroluj je, zachovej a doplň testy.

## B1. Jednotné API chyby

- přidej poslední Express error middleware;
- nezachycená chyba vrátí JSON, ne HTML;
- odpověď obsahuje bezpečnou českou hlášku a request ID;
- SQL dotaz, stack ani tajné hodnoty neposílej klientovi;
- server zaloguje request ID, method, path a plnou interní chybu.

Dotčený modul:

- `artifacts/api-server/src/app.ts`.

## B2. Jednotný opravitelný chybový stav

Vytvoř/reuse komponentu typu `QueryErrorState`:

- co se nepodařilo načíst;
- že data nebyla smazána;
- tlačítko `Zkusit znovu`;
- volitelně request ID;
- nesmí se vydávat za prázdný stav.

Použij minimálně na:

- zakázky;
- kalendář;
- knihu pohybů;
- nevyfakturované práce;
- nastavení fakturace;
- statistiky.

## B3. Kalendář nesmí zobrazit falešně prázdný den

- loading = skeleton;
- chyba = error state;
- pouze úspěšná prázdná odpověď = `Žádné zakázky na tento den`;
- query retry maximálně jeden automatický pokus, potom akce uživatele.

## B4. Neuložená zakázka

- varuj před kliknutím na interní navigaci;
- varuj před reloadem/zavřením;
- externí odkaz do nové karty neblokuj;
- po úspěšném uložení neukazuj falešné varování;
- při chybě serveru zachovej celý formulář.

Dotčené moduly:

- `job-form.tsx`;
- reusable hook `use-unsaved-changes.ts`.

## B5. Mobilní překryv ukládání

- na `/jobs/new` nezobrazuj globální FAB pro novou zakázku;
- spodní `Uložit zakázku` musí být celé klikatelné;
- bottom nav ani safe area nesmí tlačítko překrývat;
- zachovej stabilní rozměry při pending/loading stavu.

Toto není mobilní přeskupení fakturace. Fakturační rozcestník neměň.

## B6. Globální versus lokální `+`

### Problém

Na skladu, akcích nebo strojích může stejná ikona `+` znamenat jinou entitu než globální nová zakázka.

### Implementace

- globální FAB zobrazuj jen na dashboardu, kalendáři a seznamu zakázek;
- na skladu používej lokální `Přidat položku`;
- u akcí `Přidat dlouhodobou akci`;
- u strojů `Přidat stroj`;
- na desktopu i mobilu používej přístupné názvy/`aria-label`;
- neměň pořadí ani prioritu fakturačních karet.

# FÁZE C - P1 PRODUKTOVÁ FUNKČNOST

## C1. Dlouhodobá akce jako větší stavba s více výjezdy

### Cíl

Dlouhodobá akce představuje stavbu/projekt, na který se jede vícekrát. Nesmí být jen dlouhou poznámkou s jedním časovačem.

### Datový model

Implementuj `activity_visits` nebo bezpečně zobecni stávající `job_visits`.

Každý výjezd musí mít:

- ID akce;
- datum;
- volitelný čas od/do;
- přiřazeného technika;
- stav `planned | in_progress | completed | cancelled`;
- krátkou poznámku „co se dělalo“;
- volitelný další krok;
- datum vytvoření a autora;
- audit změn.

Podle existující architektury umožni vazbu výjezdu na:

- časové záznamy;
- fotografie;
- spotřebovaný materiál;
- vícepráce;
- doklady.

Pokud přímá vazba všech položek v první bezpečné iteraci výrazně zvyšuje riziko, přidej alespoň `visit_id` jako nullable a zachovej zpětnou kompatibilitu.

### UI akce

Detail dlouhodobé akce musí ukázat:

- aktuální fázi/etapu;
- další plánovaný výjezd;
- odpovědnou osobu;
- počet dokončených a plánovaných výjezdů;
- chronologii návštěv;
- rychlé `Přidat výjezd`;
- dokončení výjezdu bez dokončení celé stavby;
- dokončení celé akce pouze samostatnou akcí.

Seznam dlouhodobých akcí musí ukázat:

- zákazníka;
- stav/fázi;
- poslední návštěvu;
- další výjezd;
- odpovědného člověka;
- fakturační stav;
- varování bez zákazníka nebo bez dalšího termínu.

### Fakturace

- dokončení jednotlivého výjezdu nesmí automaticky označit celou akci jako hotovou;
- akce může být fakturována průběžně pouze s jednoznačnými source links;
- stejný materiál/práce nesmí vstoupit do dvou faktur;
- zvaž milníky/etapy pro dílčí fakturaci;
- zachovej autoritativní vazbu přes `invoice_source_links`.

### Acceptance

1. Založ akci „Rekonstrukce RD“.
2. Přidej tři výjezdy pro dva techniky.
3. Dokonči první výjezd s poznámkou, časem, fotkou a materiálem.
4. Ověř, že celá akce zůstala rozpracovaná.
5. Naplánuj další výjezd.
6. Dokonči celou akci.
7. Ověř vstup do nevyfakturovaných prací.
8. Reload nesmí změnit stav ani pořadí.

## C2. Jedna pracovní fronta pro doklady a ceny

### Problém

Fakturant přechází mezi seznamem dokladů, detailem, řádky, skladem, zakázkou a nastavením přirážek.

### Implementace

Vytvoř pracovní tabulku `K vyřízení`, která pro každý problematický řádek zobrazí:

- doklad a dodavatele;
- číslo/variabilní symbol;
- název vytěžené položky;
- množství a jednotku;
- vytěženou nákupní cenu;
- confidence;
- navrženou skladovou položku;
- navrženou zakázku nebo akci;
- starou cenu;
- novou cenu;
- procentní změnu;
- konkrétní důvod ruční kontroly.

Akce:

- upravit řádek inline;
- vybrat existující skladovou položku;
- založit novou skladovou položku;
- propojit se zakázkou/akcí;
- potvrdit jeden řádek;
- hromadně potvrdit bezpečné řádky;
- přeskočit s důvodem;
- vrátit k opravě.

Před hromadným potvrzením zobraz diff:

- kolik cen se změní;
- kolik skladových položek vznikne;
- které zakázky/akce dostanou materiál;
- které řádky zůstanou nevyřešené.

Každý propis musí být idempotentní a auditovaný.

### Dotčené moduly

- billing review queue;
- billing document detail;
- cost document service;
- material propagation;
- warehouse price history;
- audit log.

### Acceptance

- opakované potvrzení stejného řádku nevytvoří duplicitu;
- chybnou AI položku lze opravit přímo ve frontě;
- uživatel vidí starou a novou cenu před uložením;
- potvrzení jasně oznámí, co bylo změněno;
- nevyřešené řádky zůstanou viditelné.

## C3. Zdroj a historie ceny ve skladu

U každé položky ukaž:

- aktuální nákupní cenu;
- datum ceny;
- zdroj `doklad | ruční zadání | import | počáteční stav`;
- dodavatele a proklik na doklad, pokud existuje;
- předchozí cenu;
- trend změny;
- počet výdejů bez historické ceny.

Pravidla:

- změna aktuální ceny nemění cenu starých pohybů;
- doklad vytváří price-history záznam;
- ruční změna vyžaduje poznámku;
- výrazná změna ceny nad konfigurovaný práh vyžaduje potvrzení;
- položka bez ceny má konkrétní opravnou akci, ne jen červený rámeček.

## C4. Provozní zdraví aplikace pro admina

Vytvoř admin stránku bez tajných údajů:

- build SHA/verze frontendu;
- build SHA/verze API;
- očekávaná a aplikovaná DB migrace;
- stav migration parity;
- DB dostupnost;
- object storage dostupnost;
- poslední úspěšná záloha;
- poslední restore test;
- SMTP/IMAP/Gmail stav;
- AI configured/enabled/ready;
- service worker verze;
- počet posledních frontend a backend chyb.

Nezobrazuj connection string, token, heslo, API key ani bucket secret.

## C5. PWA verze a deploy drift

- frontend i API musí mít stejný release identifier;
- health/admin stránka musí odhalit rozdíl;
- update prompt se zobrazí jen při skutečně novém buildu;
- po aktualizaci se nesmí vracet ve smyčce;
- při kritické nekompatibilitě UI/API zobraz blokující provozní hlášku místo falešných dat.

# FÁZE D - P2 KOMFORT A VÝKON

## D1. Rozdělení nastavení

Rozděl dlouhou stránku nastavení do záložek:

- Vzhled;
- Firma a dokumenty;
- E-mail;
- Import dokladů;
- Zálohy;

Pravidla:

- deep link do záložky je možný;
- technický text zobraz jen adminovi;

## D2. Zhutnění mobilního skladu

Na šířce 390 px:

- název maximálně dva řádky;
- stav skladu a cena mají stabilní místo;
- hlavní akce je příjem/výdej;
- editace, historie a smazání jsou v menu;
- význam ikon má tooltip/aria-label;
- nedochází k horizontálnímu overflow;
- globální FAB nekoliduje s lokální akcí.

Zachovej na desktopu rychlé skenování:

- název/SKU;
- stav;
- nákupní/prodejní cena;
- zdroj ceny;
- minimum;
- poslední pohyb.

## D3. Výkon prvního načtení

Aktuální build měl přibližně:

- hlavní JS přes 3 MB před gzip;
- HEIC chunk přes 1,3 MB.

Implementuj route-level lazy loading pro:

- fakturaci;
- PDF;
